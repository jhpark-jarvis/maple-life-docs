# Codex 작업 가이드

> 운영 원칙
> 이 문서는 에이전트가 작업 순서와 해석 규칙을 빠르게 잡기 위한 정적 작업 가이드입니다.
> 이 문서만으로 모든 세부 구현을 확정하지 말고, 새로운 작업이나 모호한 작업은 `<MAPLE_LIFE_DOCS_BASE_URL>` API 또는 D1 원문으로 반드시 보조 확인해야 합니다.
> `<MAPLE_LIFE_DOCS_BASE_URL>`는 문서 내 placeholder이며, 실제 값은 에이전트 실행 시 환경 변수 또는 secret으로 주입되어야 합니다.

## 1. 이 문서의 역할

이 문서는 Codex가 Maple Life 관련 개발/기획 작업을 진행할 때 어떤 기준으로 문서를 읽고, 어떤 방향으로 구현 판단을 내려야 하는지 정리한 작업 규칙 문서다.

원본은 D1에 있고, 이 문서는 작업 효율을 위한 압축본이다.

## 2. 권장 작업 방식

권장 구조는 아래와 같다.

### 2-1. 원본 소스

- `Cloudflare D1 documents`

### 2-2. 개발 실행 기준

- `MSW MCP`가 제공하는 공식 skills와 공식 문서
- `MSW MCP`가 제공하는 API, 오브젝트 모델, 맵 편집/배치 방식, 엔진 제약

위 항목이 개발 시 최우선 실행 기준이다.

### 2-3. 작업용 컨텍스트

- `.docs/msw/msw-project-overview.md`
- `.docs/msw/codex-working-guide.md`
- `.docs/msw/maple-life-docs-api-reference.md`

### 2-4. 세부 검증

아래 상황에서는 D1 원문을 다시 확인한다.

- 새로운 작업을 처음 시작할 때
- 현재 가이드만으로 해석이 모호할 때
- 새 시스템을 구현할 때
- 기획 명칭이 바뀌었을 가능성이 있을 때
- 초안/작성 예정 문서의 실제 진행 상태를 확인해야 할 때
- 데이터 테이블 문서와 본문 문서가 충돌할 때

즉, 평소에는 MSW MCP의 실행 기준 위에서 가이드를 보고 작업하고, 분기점에서만 D1 원문을 다시 조회하는 방식이 가장 효율적이다.

### 2-5. 조회 방식 규칙

- 기본은 `.docs/msw/` 정적 문서를 먼저 읽는다.
- 아래 조건이면 관련 D1 문서를 API 또는 repository 경로로 다시 조회한다.
  - 작업이 완전히 새롭다
  - 요구사항이 모호하다
  - 문서 간 충돌 가능성이 있다
  - 최근 변경 여부가 중요하다
  - 초안 데이터를 실제 구현에 반영하려고 한다

즉, `static guide first, API verification on ambiguity or new work` 원칙을 유지한다.

### 2-6. API base URL 규칙

문서에서 말하는 API 조회는 현재 작업 대상이 되는 `maple-life-docs` 앱을 기준으로 한다.

- 기본 운영 API base URL: `<MAPLE_LIFE_DOCS_BASE_URL>`
- 기본 API prefix: `/api`
- 문서 보조 엔드포인트 prefix: `/documents`

base URL 규칙:

- 별도 지시가 없으면 `<MAPLE_LIFE_DOCS_BASE_URL>`를 기본 조회 대상으로 사용한다.
- 즉, 에이전트는 현재 운영 프론트가 실제로 사용 중인 API를 기준으로 문서와 데이터를 확인한다.
- 다른 환경이 명시적으로 주어지면 그 base URL로 바꾼다.
- 문서만으로 실제 host를 확정하지 않고, 주입값이 없으면 사용자 또는 실행환경에서 받아야 한다.
- `localhost`나 임의 도메인을 추정하지 않는다.

예:

- `<MAPLE_LIFE_DOCS_BASE_URL>/api/documents`
- `<MAPLE_LIFE_DOCS_BASE_URL>/api/documents/{document_id}`
- `<MAPLE_LIFE_DOCS_BASE_URL>/api/dashboard`
- `<MAPLE_LIFE_DOCS_BASE_URL>/documents/search-links?q=...`

### 2-7. 조회 경로 우선순위

에이전트는 아래 순서로 조회 경로를 선택한다.

1. 정적 가이드 문서 `.docs/msw/*`
2. API 목록 확인용 `.docs/msw/maple-life-docs-api-reference.md`
3. 앱 공개 API
4. 숨김 문서/최신 원문 검증이 필요하면 repository 또는 D1 direct query

실행 전제:

- 에이전트는 먼저 기본 운영 base URL이 `<MAPLE_LIFE_DOCS_BASE_URL>`라는 사실을 따른다.
- 다른 base URL이 명시적으로 주어지면 그 값으로 교체한다.
- `<MAPLE_LIFE_DOCS_BASE_URL>` 값 자체는 이 문서에 포함되지 않으며, 실행 시점에 외부에서 주입된 값을 사용한다.

공개 API 예시:

- 문서 목록: `GET /api/documents`
- 문서 상세: `GET /api/documents/{document_id}`
- 대시보드 맥락: `GET /api/dashboard`
- 문서 링크 검색: `GET /documents/search-links?q=...`

주의:

- `GET /api/documents`는 화면용 목록이므로 숨김 문서 포함 여부를 확인해야 한다.
- 숨김 문서까지 포함한 최신 원문 검증은 공개 API만으로 부족할 수 있으므로 D1 repository 또는 direct query를 사용한다.

## 4. 구현 판단 기준

Codex가 기능/기획 판단을 할 때 아래 기준을 우선 적용한다.

### 4-0. 실행 방식 기준

- 구현에 들어가기 전에 관련 `MSW MCP` 공식 skills와 문서를 먼저 확인한다.
- 구현 방법은 `MSW MCP`가 제공하는 모델과 제약을 우선 따른다.
- 우리 문서는 “무엇을 만들어야 하는지”, “어떤 경험을 내야 하는지”, “무엇을 우선해야 하는지”를 정의한다.
- 즉, `설계 의도는 우리 문서`, `실행 해상도는 MSW MCP`라는 역할 분리를 유지한다.
- API 조회가 필요하면 기본적으로 `<MAPLE_LIFE_DOCS_BASE_URL>`에 붙고, 별도 환경이 주어졌을 때만 교체한다.
- `MSW` 기본 기능/기본 자산으로 처리할지, `커스텀 구현` 또는 `커스텀 자산`으로 갈지 선택지가 열려 있으면 에이전트가 독단적으로 정하지 않고 반드시 사용자에게 먼저 질문한 뒤 작업한다.

### 4-1. 주민 생활성 우선

아래 방향이 맞다.

- 생활
- 상점 운영
- 경제 활동
- 관계 형성
- 지역 정착
- 수집과 기록

아래 방향은 주력 축이 아니다.

- 전투 스펙 경쟁 중심
- 몬스터 사냥 위주 루프
- 영웅 서사 중심 진행

### 4-2. 콘텐츠는 루프로 봐야 한다

문서를 읽을 때 단건 기능보다 루프를 우선 해석한다.

예시:

- 잡화상점은 “포션 제조 UI”가 아니라
  - 준비
  - 주문
  - 제작
  - 판매
  - 정산
  - 성장
  - 재투자의 루프다.

### 4-3. 경제는 역할 분리를 유지해야 한다

새 재화나 비용 구조를 제안할 때는 다음을 확인한다.

- 메소와 콘텐츠 전용 재화가 같은 역할을 하지 않는가
- 소비처가 성장 루프와 연결되는가
- 자동 수익 장치가 아닌 플레이 기반 보상인가

### 4-4. 맵은 미관보다 목적 적합성이 중요하다

맵/리소스 작업에서는 다음을 본다.

- 이 공간이 필드형인지
- 실내 배치형인지
- 횡스크롤 액션형인지
- 어떤 TileMap이 맞는지
- 플레이 동선과 상호작용 포인트가 분명한지

## 5. 초안 문서 해석 규칙

현재 D1에는 `(초안)` 표기가 붙은 문서와 데이터 테이블 문서가 있다.

예:

- `14-01-34. CustomerStorage (초안)`
- `14-01-36. Upgrade Storage(초안)`

이런 문서는 아래처럼 해석한다.

- 시스템 방향은 참조 가능
- 수치는 확정값으로 단정하지 않는다
- 실제 구현 전에는 별도 검증이 필요하다
- 상위 개념 문서와 충돌하면 상위 개념 문서를 우선한다

## 6. 문서 충돌 시 원칙

문서 내용이 충돌하거나 명칭이 바뀐 흔적이 보이면 아래 순서로 판단한다.

1. 최근 수정된 문서를 우선 확인
2. 상위 문서와 하위 문서 관계를 확인
3. 개념 문서와 데이터 문서를 구분
4. 최종 확정이 없으면 “미확정”으로 표시하고 사용자 확인 포인트로 남긴다

## 7. 추천 운영 방식

향후 Codex 또는 MCP 기반 개발 시 운영 방식은 아래가 가장 좋다.

1. D1 문서를 정기적으로 읽어 가이드 문서를 갱신
2. 구현 작업은 `MSW MCP` 공식 skills/문서와 실행 모델을 기준으로 진행
3. 기능 방향과 우선순위는 `.docs/msw/` 문서를 우선 컨텍스트로 사용
4. 큰 기능 단위, 새로운 작업, 모호한 작업에서는 관련 D1 원문 문서만 추가 조회
5. 구현 후 변경사항이 있으면 다시 D1 문서 또는 `.docs/msw/` 가이드에 반영
6. `MSW 기본 방식`과 `커스텀 방식`이 모두 가능한 분기에서는 먼저 사용자 선택을 받고 그 기준으로 구현 방향을 확정

## 8. 지금 시점 핵심 결론

현재 기준으로 Codex가 가장 먼저 이해해야 하는 것은 아래다.

- Maple Life는 메이플 월드 주민 생활 RPG다.
- 가장 구체적인 핵심 콘텐츠는 헤네시스 잡화상점이다.
- 맵 제작은 MSW TileMap 유형 구분이 매우 중요하다.
- 하우징, 주민센터, 가구, 도감은 생활 확장축이다.
- 문서 최신 원본은 D1이며, 개발 실행은 MSW MCP 기준으로 맞추고, 작업용 컨텍스트는 별도 가이드 문서로 유지하는 편이 가장 효율적이다.
