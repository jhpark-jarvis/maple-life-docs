# MSW Agent Optimization Rules

> 운영 원칙
> 이 문서는 MSW 개발용 에이전트의 공용 규칙 문서이며, 정적 기준축을 제공합니다.
> 최신 원문 전체를 대체하지 않으므로, 새 작업 시작, 모호한 요구사항, 최신성 검증이 필요한 경우에는 `<MAPLE_LIFE_DOCS_BASE_URL>` API 또는 D1 원문을 다시 조회해야 합니다.

## 1. 목적

이 문서는 `MSW MCP`에 연결된 에이전트가 `maple-life-docs` 저장소 안의 문서, 코드, 기존 구현 자산을 최대한 효율적으로 활용해 일관되고 품질 높은 결과를 내도록 만드는 공용 운영 규칙이다.

이 문서의 목표는 3가지다.

- 최신 기획 원본과 구현 자산을 함께 활용하게 한다.
- 개발 실행은 `MSW MCP` 기준으로, 기획 방향과 구현 포인트는 `maple-life-docs` 문서 기준으로 역할을 분리한다.
- 개발 시 가장 먼저 참고해야 할 것은 `MSW MCP`가 제공하는 공식 skills와 공식 문서라는 점을 명확히 한다.
- 에이전트가 임의 해석으로 과도하게 벗어나는 것을 막는다.
- 결과물을 바로 구현 가능한 수준의 작업 산출물로 유도한다.

## 2. 진실의 원천 우선순위

에이전트는 아래 우선순위로 정보를 해석한다.

### 2-1. 개발 실행 기준

가장 먼저 따라야 하는 실행 기준은 `MSW MCP`가 제공하는 기능과 제약이다.

- 공식 skills
- 공식 문서
- 실제 사용 가능한 툴셋
- 맵/오브젝트 편집 방식
- 엔진 제약
- 자동화 가능한 범위
- 생성 가능한 산출물 포맷

에이전트는 구현 방법을 제안할 때 반드시 이 실행 기준 안에서 판단해야 하며, 가능하면 먼저 관련 `MSW MCP` 공식 skills와 문서를 확인해야 한다.

### 2-2. 기획 원본

가장 우선하는 원본은 `Cloudflare D1 documents`다.

- 프로젝트 방향
- 세계관
- 시스템 정의
- 콘텐츠 세부 기획
- 맵 제작 기준
- 데이터 테이블 초안

문서 원본과 가이드 문서가 충돌하면 D1 원문을 우선한다.

### 2-3. 작업용 압축 가이드

에이전트는 매 작업 시작 시 아래 문서를 우선 읽는다.

- `.docs/msw/msw-project-overview.md`
- `.docs/msw/codex-working-guide.md`
- `.docs/msw/maple-life-docs-api-reference.md`
- 현재 문서인 `.docs/msw/msw-agent-optimization-rules.md`

이 문서들은 D1 원문 전체를 매번 다시 읽지 않고도 방향성을 빠르게 잡기 위한 압축본이다.

기본 원칙은 아래와 같다.

- 먼저 정적 가이드를 읽는다.
- 어떤 API가 있는지 감이 필요하면 `maple-life-docs-api-reference.md`를 먼저 확인한다.
- 모호한 부분이 있거나 새로운 작업을 시작하면 관련 D1 원문을 API 또는 repository 경로로 재조회한다.
- `MSW` 기본 기능/기본 자산과 `커스텀 구현`/`커스텀 자산`이 모두 가능한 경우에는 에이전트가 임의로 선택하지 않고 반드시 사용자 확인을 먼저 받는다.

### 2-3-1. API 위치 규칙

이 문서에서 말하는 API는 현재 작업 대상이 되는 `maple-life-docs` 앱의 HTTP API다.

- 기본 운영 API base URL: `<MAPLE_LIFE_DOCS_BASE_URL>`
- 기본 API prefix: `/api`
- 문서 보조 prefix: `/documents`

base URL 규칙:

- 별도 지시가 없으면 `<MAPLE_LIFE_DOCS_BASE_URL>`를 기본 조회 대상으로 사용함
- 즉, MSW 개발용 에이전트도 우선 현재 운영 프론트가 실제로 사용하는 API를 기준으로 확인함
- 다른 base URL이 명시적으로 주어졌을 때만 교체함
- `localhost`나 임의 host를 추정하지 않음

대표 경로:

- `GET <MAPLE_LIFE_DOCS_BASE_URL>/api/documents`
- `GET <MAPLE_LIFE_DOCS_BASE_URL>/api/documents/{document_id}`
- `GET <MAPLE_LIFE_DOCS_BASE_URL>/api/dashboard`
- `GET <MAPLE_LIFE_DOCS_BASE_URL>/documents/search-links?q=...`

### 2-4. 실제 구현 자산

기획을 코드로 바꾸는 단계에서는 아래를 확인한다.

- `app/`
  - API, 저장소 구조, 데이터 흐름, 라우팅
- `frontend/src/`
  - 현재 UI 구조, 라우팅, 상태 관리 방식, 컴포넌트 패턴
- `worker-python/`
  - Worker 기반 이전 시도 및 API 대응 자산

즉, `실행 방식은 MSW MCP`, `기획 의도는 D1 원문`, `현재 구현 제약은 저장소 코드`가 기준이다.

## 3. 에이전트 기본 해석 규칙

에이전트는 Maple Life를 아래처럼 해석해야 한다.

- 메이플 월드 주민 생활 RPG다.
- 영웅 서사보다 생활 루프가 우선이다.

### 3-2. 설계 우선순위

아래를 우선한다.

- 생활
- 경제
- 관계
- 지역 체험
- 직업 루프
- 하우징
- 수집/기록

아래는 보조 축이다.

- 전투 경쟁
- 사냥 효율
- 스펙 중심 성장

### 3-3. 결과물 해석 기준

에이전트는 기능을 단건 위젯으로 보지 말고 루프로 해석해야 한다.

예시:

- 잡화상점 = 포션 제작 UI가 아니라
  - 준비
  - 주문
  - 제작
  - 판매
  - 정산
  - 성장
  - 재투자의 반복 루프

## 4. 작업 시작 절차

에이전트는 새 작업마다 아래 절차를 따른다.

1. 작업 유형을 분류한다.
2. 관련 `MSW MCP` 공식 skills와 문서가 있는지 먼저 확인한다.
3. 관련 문서 우선순위를 결정한다.
4. 기존 코드/자산 재사용 가능성을 확인한다.
5. 결과물 수준을 정한다.
6. 부족한 정보가 있으면 “미확정 항목”으로 분리한다.
7. 작업이 새롭거나 모호하면 관련 D1 문서를 API로 다시 조회한다.
8. `MSW 기본 방식`과 `커스텀 방식`이 함께 가능한 분기가 있으면 사용자 선택을 먼저 받는다.

작업 시작 전에 항상 아래 두 질문을 먼저 정리한다.

1. 이 작업을 `MSW MCP` 안에서 실제로 어떤 방식으로 구현할 수 있는가?
2. 그 구현이 `maple-life-docs` 문서가 요구하는 경험과 방향을 만족하는가?
3. 필요한 원문 검증은 공개 API로 충분한가, 아니면 D1 repository/direct query가 필요한가?
4. 현재 API를 기본 운영 base URL(`<MAPLE_LIFE_DOCS_BASE_URL>`)로 조회하면 충분한가?

## 5. 자산 활용 규칙

에이전트는 결과를 만들 때 기존 자산을 우선 재사용해야 한다.

### 5-1. 기존 자산 우선

아래 순서로 재사용 가능성을 확인한다.

1. 현재 프로젝트 코드 구조
2. 기존 문서에 정의된 오브젝트/루프/명칭
3. 기존 MSW 타일맵 기준
4. 기존 리소스 문서
5. 없는 경우 placeholder 또는 신규 필요 자산으로 분리

### 5-2. 금지 규칙

아래는 임의 확정하면 안 된다.

- 최종 아트 스타일
- 새 세계관 설정
- 확정되지 않은 수치 밸런스
- 존재하지 않는 경제 재화
- 문서에 없는 시스템 축
- `MSW` 기본 방식으로 갈지 `커스텀 구현/커스텀 자산`으로 갈지에 대한 분기 선택

### 5-3. 허용 규칙

아래는 조건부 허용이다.

- placeholder 오브젝트 정의
- AI 리소스 초안 제안
- 미확정 항목 분리 보고
- 구현 가능한 범위의 합리적 기본값 제안

단, 이 경우 반드시 아래 중 하나로 표시한다.

- `가정`
- `초안`
- `추후 확정 필요`

### 5-4. 사용자 확인이 필요한 분기

아래 상황은 에이전트가 스스로 확정하면 안 된다.

- `MSW` 기본 기능으로도 구현 가능하고, 별도 `커스텀 기능`으로도 구현 가능한 경우
- `MSW` 기본 자산으로도 대응 가능하고, 별도 `커스텀 자산` 제작으로도 대응 가능한 경우
- 일정, 품질, 유지보수 방식이 달라지는 구현 분기인데 문서만으로 우선순위가 확정되지 않은 경우

이 경우 에이전트는 반드시 아래 흐름을 따른다.

1. 현재 가능한 선택지를 짧게 정리한다.
2. 각 선택지의 차이점과 영향 범위를 설명한다.
3. 사용자에게 어떤 방향으로 갈지 질문한다.
4. 사용자 답변을 받은 뒤에만 실제 구현을 진행한다.

## 6. 코드 활용 규칙

에이전트는 기획만 보지 말고 현재 구현 상태도 반드시 확인한다.

단, 현재 저장소 코드는 참고 자산일 뿐이며, MSW 실제 구현 방식과 충돌하면 `MSW MCP` 제약을 우선한다.

### 6-1. 백엔드 확인 포인트

- `app/api.py`
  - 현재 API 표면
- `app/repositories/`
  - 데이터 접근 구조
- `app/frontend.py`
  - 프론트 라우팅 연결 방식
- `app/storage.py`
  - 자산 저장 방식

### 6-2. 프론트 확인 포인트

- `frontend/src/App.jsx`
  - 라우팅 구조
- `frontend/src/pages/`
  - 페이지 구성
- `frontend/src/components/`
  - 재사용 컴포넌트

### 6-3. 구현 판단 원칙

- 기존 구조를 깨지 않고 들어갈 수 있으면 확장
- 반복되는 패턴이 이미 있으면 그대로 재사용
- 새 추상화는 기존 구조로 감당이 안 될 때만 도입

## 8. 미확정 항목 처리 규칙

에이전트는 애매한 내용을 임의 확정하지 말고 분리해야 한다.

아래는 반드시 분리 보고한다.

- 문서 간 상충되는 명칭
- 초안 수치
- 신규 자산 필요 여부
- 최종 아트 방향
- 시스템 범위 확장 여부

보고 형식은 짧고 명확해야 한다.

예시:

- `미확정: 주민센터 금융 기능이 실제 플레이 가능 콘텐츠인지 허브 기능인지 추가 확인 필요`
- `미확정: 상점 전용 토큰 명칭과 획득 공식은 아직 확정되지 않음`

미확정 항목이 실제 작업 방향을 바꿀 수 있는 수준이면, 관련 D1 원문을 다시 조회한 뒤 판단한다.

## 9. 최대 효율 운용 전략

에이전트가 효율을 높이려면 아래 원칙을 지켜야 한다.

### 9-1. 매번 전체 원문을 다시 읽지 않는다

기본은 압축본을 읽고 시작한다.

- `.docs/msw/msw-project-overview.md`
- `.docs/msw/codex-working-guide.md`
- `.docs/msw/msw-agent-optimization-rules.md`

그리고 먼저 `MSW MCP` 실행 가능 범위를 확인한 뒤, 아래 조건일 때만 특정 기능과 직접 연결된 D1 원문을 API로 추가 조회한다.

- 완전히 새로운 작업 시작
- 요구사항 해석이 모호함
- 문서 간 충돌 가능성 존재
- 최근 변경 여부가 중요함
- 초안 수치/초안 구조를 실제 구현에 반영하려고 함

공개 API만으로 충분한 경우:

- 공개 문서 목록 확인
- 특정 공개 문서 상세 확인
- 현재 화면이 어떤 데이터를 쓰는지 확인

repository/D1 direct query가 필요한 경우:

- 숨김 문서까지 포함한 원문 확인
- 최신 수정분 검증
- 공개 API가 의도적으로 제외한 데이터 확인

### 9-2. 결과물은 항상 “바로 다음 작업 가능 상태”여야 한다

좋은 결과물:

- MSW MCP 안에서 바로 만들 수 있는 작업 계획
- 바로 구현 가능한 작업 지시서
- 바로 맵 제작 가능한 배치안
- 바로 API 수정 가능한 변경 지시

나쁜 결과물:

- 추상적인 감상
- 방향만 있고 금지사항이 없는 문서
- 자산 정책 없이 아트 해석을 넘겨버리는 설계안

### 9-3. 결과는 재사용 가능해야 한다

가능하면 결과를 아래 중 하나로 남긴다.

- 공용 가이드 문서
- 작업 지시서 템플릿
- 구현 체크리스트
- 자산 필요 목록

## 10. 다른 에이전트에도 이식 가능한 공용 규칙

이 문서에서 agent-agnostic 하게 재사용 가능한 핵심은 아래다.

- MSW MCP 실행 기준 우선
- D1 원문 우선
- 압축 가이드 우선 참조
- 생활형 주민 RPG 해석 우선
- 루프 기반 설계
- 기존 자산 우선 재사용
- 미확정 항목 분리
- 결과물 강제 형식 유지

즉, 실행 포맷은 각 agent마다 다를 수 있지만, 판단 규칙은 공용화 가능하다.

## 11. 붙여 넣기용 시스템 프롬프트 초안

아래 블록은 MSW MCP 에이전트에 바로 전달할 수 있는 운영 프롬프트 초안이다.

```text
너는 Maple Life 프로젝트를 지원하는 구현/기획 에이전트다.

프로젝트 해석 규칙:
- Maple Life는 메이플스토리 월드 기반의 생활형 주민 RPG다.
- 플레이어는 영웅이 아니라 주민이다.
- 전투 효율보다 생활 루프, 경제, 관계, 지역 체험, 하우징, 수집을 우선한다.
- 기능을 단건 UI가 아니라 플레이 루프로 해석한다.

역할 분리:
- 개발 실행 방식은 MSW MCP가 제공하는 기능과 제약을 기준으로 한다.
- 무엇을 만들어야 하는지와 어떤 경험을 내야 하는지는 maple-life-docs 문서를 기준으로 한다.

정보 우선순위:
1. MSW MCP 공식 skills, 공식 문서, 실행 가능 범위와 제약
2. Cloudflare D1 문서 원본
3. .docs/msw/ 가이드 문서
4. maple-life-docs 현재 코드와 구현 자산

작업 규칙:
- API 조회는 기본적으로 `<MAPLE_LIFE_DOCS_BASE_URL>`를 사용한다.
- 다른 환경이 명시적으로 주어졌을 때만 base URL을 교체한다.
- 작업 시작 시 관련 `MSW MCP` 공식 skills와 문서가 있다면 그것부터 먼저 참고한다.
- 먼저 이 작업을 MSW MCP 안에서 어떤 방식으로 실현할 수 있는지 정리한다.
- 먼저 관련 문서를 좁혀서 읽고 핵심 근거를 정리한다.
- `MSW` 기본 기능/자산과 `커스텀 구현`/`커스텀 자산`이 모두 가능한 경우에는 반드시 사용자에게 먼저 선택을 요청한다.
- 기존 코드/자산/명칭을 우선 재사용한다.
- 없는 자산은 임의 확정하지 말고 placeholder 또는 신규 필요 자산으로 분리한다.
- 초안 수치와 미확정 내용은 확정값처럼 쓰지 않는다.
- 결과는 항상 바로 다음 구현 작업이 가능한 형태로 정리한다.

출력 형식:
- 작업 목표
- 핵심 컨셉
- 문서 근거
- 구현 방향
- 금지사항
- 미확정 항목
- 산출물

맵/콘텐츠 작업 추가 규칙:
- 공간의 역할, 플레이 루프, TileMap 선택 이유를 반드시 설명한다.
- NPC는 기능 안내문이 아니라 세계 안 주민처럼 설계한다.
- 오브젝트와 자산은 기존 문서/기존 자산 재사용을 우선한다.
```

## 12. 추천 사용 방식

가장 추천하는 방식은 아래다.

1. 에이전트 시작 시 이 문서를 읽힌다.
2. `MSW MCP` 실행 가능 범위를 먼저 확인시킨다.
3. `.docs/msw/msw-project-overview.md`를 함께 읽힌다.
4. 작업 대상 기능에 맞는 D1 원문만 추가 조회한다.
5. 산출물은 이 문서의 형식으로 강제한다.

이렇게 하면 토큰 낭비를 줄이면서도, 프로젝트 방향을 잃지 않고 높은 품질의 결과를 반복해서 얻기 쉽다.
